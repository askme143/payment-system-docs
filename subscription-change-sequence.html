Sequence Diagram

구독 플랜 변경 플로우

회원이 현재 상품 구독 안에서 플랜을 변경할 때 즉시 업그레이드는 차액을 청구하고, 다운그레이드나 다음 주기 변경은 예약 상태로 저장한 뒤 다음 정기 결제부터 반영하는 흐름입니다.

전체 시퀀스 목록전체 API 목록API 상세 설명이메일 발송 시스템 아키텍처 문서MongoDB 구조 문서결제 시스템 잠재 위험 및 대응 문서

사용 API

GET /plans

구매 가능한 상품과 구독 플랜 목록을 조회합니다.

GET /plans/{planId}

단일 플랜의 가격, 주기, 판매 가능 상태를 조회합니다.

POST /subscriptions/{subscriptionId}/change-preview

목표 플랜 변경이 업그레이드인지 다운그레이드인지 서버가 판정하고, 유저 확인에 필요한 결제 금액과 적용일을 계산합니다.

PATCH /subscriptions/{subscriptionId}

플랜 변경 미리보기에서 발급한 확인 토큰으로 업그레이드는 즉시 변경하고 다운그레이드는 다음 결제일 반영으로 예약합니다.

GET /subscriptions/me

현재 회원의 상품별 구독 목록, 이용 기간, 다음 결제일, 공통 기본 결제수단 표시 정보를 조회합니다.

POST /internal/subscription-billing/run

다음 결제일이 도래한 구독을 찾아 정기 자동결제를 실행합니다.

POST /webhooks/toss-payments

토스페이먼츠 결제 이벤트를 수신하고 내부 결제/인보이스/구독 상태를 보정합니다.

연관 잠재 위험
  • provider-paid-internal-unpaid PG 승인 성공 후 내부 미반영
  • plan-change-double-apply 플랜 변경 중복 실행
  • duplicate-active-subscription 상품별 활성 구독 중복 생성

상태 요약

시점구독 상태결제 상태설명
변경 요청 전activepaid회원은 현재 플랜으로 이용 중이며 저장된 빌링키가 활성 상태입니다.
차액 결제 생성 후changing_planpending중복 변경을 막기 위해 변경 작업과 결제 회차가 잠깁니다.
차액 결제 성공 후activepaid목표 플랜이 즉시 적용되고 다음 정기 결제 금액도 새 플랜 기준으로 갱신됩니다.
변경 예약 전activepaid현재 플랜 권한과 다음 결제일이 유지됩니다.
변경 예약 직후activepaidpendingPlanId와 effectiveAt이 저장되고 현재 기간에는 기존 플랜이 유지됩니다.
다음 정기 결제 시activepending예약된 플랜이 실제 planId로 승격되고 새 플랜 금액으로 청구가 시작됩니다.
