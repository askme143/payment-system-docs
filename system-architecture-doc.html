개요

이메일 알림 outbox가 인증, 결제, 구독 상태 변경과 분리되어 큐 적재, 템플릿 인자 검증, SMTP 발송, 재시도와 dead-letter 처리를 수행하는 구조입니다.

이메일 실패는 결제, 인보이스, 구독, 감사 로그 상태를 롤백하지 않습니다. 발송 실패는 outbox 상태와 worker summary로 추적합니다.

이메일 알림 Outbox 아키텍처

결제 API와 구독 배치가 메일을 직접 보내지 않고 MongoDB outbox에 발송 의도를 저장한 뒤 Notification Worker가 비동기로 발송합니다.

컴포넌트 책임

컴포넌트책임실패 정책
Payment API / Subscription Use Cases결제, 인보이스, 구독 상태를 변경하고 메일 발송 intent를 idempotency_key와 함께 저장합니다.메일 provider 장애를 직접 처리하지 않고 outbox 저장 성공까지만 책임집니다.
Admin/User Auth ServiceOTP, 로그인 링크, 비밀번호 재설정 토큰을 생성하고 해시, 만료 시각, 인증 요청 상태를 저장한 뒤 인증 메일 intent를 적재합니다.Email Worker에 인증 판단을 위임하지 않습니다. 토큰 생성, 검증, 폐기, 재발급 정책은 Auth Service가 소유합니다.
notification_outbox발송 대상, recipient_type, template_key, template_args, expires_at, status, attempt_count, available_at, locked_until_at을 보관합니다.중복 enqueue는 deterministic idempotency_key unique index로 억제하고, expires_at이 지난 인증 메일은 발송하지 않습니다.
notification_templates목적별, 상품별, 상품 유형별, 기본 event template과 version, required_args schema를 관리합니다.템플릿 미존재 또는 template_args 검증 실패는 dead_letter로 분류합니다.
Notification Workerdue item을 claim하고 만료 여부와 template_args를 검증한 뒤 SMTPEmailSender로 전송합니다.일시 실패는 retry_scheduled, 영구 실패와 만료된 인증 메일은 dead_letter로 상태를 기록합니다.
SMTPEmailSender렌더링된 subject/html/text를 SMTP 서버에 전달하고 provider_message_id를 반환합니다.timeout, 429, 5xx는 retryable 오류로, 잘못된 수신자나 4xx는 permanent 오류로 변환합니다.

운영과 실패 처리

Worker run summary

selected_count, claimed_count, sent_count, retry_scheduled_count, dead_letter_count, skipped_count, failed_count를 기록합니다.

Retry policy

일시 실패는 1분, 5분, 30분, 2시간, 12시간 backoff 후 retry_scheduled 상태로 재시도합니다.

Dead-letter handling

dead_letter는 자동 재시도하지 않고 Mongo 상태, worker summary, 로그로 운영자가 확인합니다.

Stale processing recovery

locked_until_at이 지난 processing item은 다른 worker가 다시 claim할 수 있습니다.

Template failure visibility

template_key 누락, required template_args 누락, render 실패는 last_error와 함께 dead_letter로 남깁니다.

Auth email security boundary

OTP, 로그인 링크, 비밀번호 재설정 토큰은 Auth Service가 생성하고 검증합니다. Email Worker는 인증 메일을 발송만 하며 만료된 expires_at 이후에는 보내지 않습니다.

Template argument contract

각 template_key는 required_args schema를 갖고 enqueue 시점과 render 시점에 template_args를 검증합니다. 예: admin_auth.login_otp는 admin_name, otp_code, expires_minutes를 요구합니다.
