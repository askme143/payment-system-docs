개요

결제 시스템을 배포하고 운영하는 담당자가 API 서버, Kubernetes CronJob scheduler, Notification Worker, MongoDB, Toss 연동, 어드민 콘솔을 같은 기준으로 배포, 검증, 관측, 복구하기 위한 운영 런북입니다.

대상 독자

  • 배포를 수행하는 백엔드 엔지니어와 플랫폼 운영자
  • 결제 장애를 1차 확인하는 온콜 담당자
  • 어드민 콘솔과 스케줄러 수동 실행을 운영하는 관리자

포함 범위

  • payments 이미지 빌드와 API 서버 배포
  • Kubernetes CronJob 기반 정기결제, 사전 알림, 해지 예약 만료 배치 배포
  • 상시 실행 Notification Worker 배포와 dead-letter 확인
  • MongoDB 인덱스, 주요 컬렉션, 운영 관측 포인트
  • 어드민 콘솔의 수동 실행과 감사 로그 확인 절차

제외 범위

  • Toss 대시보드 자체 설정 절차
  • 회원/관리자 원본 서비스 배포 절차
  • 클러스터, 네트워크, 인증서 같은 플랫폼 공통 인프라 구축

런타임 구성

API 서버와 Notification Worker는 장기 실행 workload이고, scheduler batch는 같은 이미지를 Kubernetes CronJob에서 1회 실행 entrypoint로 호출합니다.

해지 예약 만료 CronJob 배포python -m payments.scheduler.runner cancel-expiration --batch-size 100cancelAt이 지난 cancel_scheduled 구독을 canceled로 전환합니다.scheduler_run_logs job_type=cancel_expiration과 관련 operator_audits를 확인합니다.

배포 전 체크리스트

  • 이미지 tag 또는 digest가 변경 승인된 revision을 가리키는지 확인합니다.
  • PAYMENTS_DATABASE_URL, PAYMENTS_DATABASE_NAME, PAYMENTS_INTERNAL_SERVICE_TOKEN이 API 서버와 worker 계열 workload에 모두 주입되어 있는지 확인합니다.
  • 운영 Toss secret, webhook secret, client key가 staging 값과 섞이지 않았는지 확인합니다.
  • PAYMENTS_ADMIN_AUTH_LINK_BASE_URL이 실제 어드민 콘솔 도메인을 가리키는지 확인합니다.
  • MongoDB index 생성 권한이 있는 계정으로 API 서버 또는 worker가 시작되는지 확인합니다.
  • CronJob command가 billing, reminder, cancel-expiration 중 하나이며 --batch-size가 운영 한도 안에 있는지 확인합니다.
  • rollback에 사용할 직전 정상 이미지 tag 또는 digest를 기록합니다.

배포 후 검증

  • API 서버 health endpoint가 200을 반환하는지 확인합니다.
  • /admin/console/login이 text/html로 렌더링되는지 확인합니다.
  • 어드민 로그인 후 /admin/console/scheduler-runs, /admin/console/operator-audits, /admin/console/products에 접근할 수 있는지 확인합니다.
  • 작은 batch 또는 --dry-run으로 scheduler runner를 실행해 scheduler_run_logs 생성 여부를 확인합니다.
  • scheduler_run_logs.status=succeeded이면서 summary.failed > 0인 실행은 일부 대상 실패로 분류하고, status=failed인 실행은 실행 자체 실패로 분류합니다.
  • notification_outbox의 pending, retry_scheduled, dead_letter 상태 분포를 확인합니다.
  • operator_audits에 운영자 수동 실행 또는 상품/결제/구독 운영 액션의 previousState와 nextState가 남는지 확인합니다.

운영 런북

CronJob 실행 자체 실패

증상

  • scheduler_run_logs.status=failed
  • exit_code=1
  • Kubernetes Job pod가 non-zero exit

확인

  • scheduler_run_logs.error.code와 request 시간대를 확인합니다.
  • Pod log에서 환경변수 누락, Mongo 연결 실패, Toss provider 오류를 확인합니다.
  • operation_locks에 같은 job lock이 stale 상태로 남았는지 확인합니다.

조치

  • 환경변수 또는 secret 누락이면 배포 manifest를 수정하고 새 Job을 실행합니다.
  • 일시 provider 장애이면 재시도 가능 시간에 CronJob 재실행 또는 어드민 수동 실행을 사용합니다.
  • stale lock이 원인이면 locked_until_at 경과 여부를 확인한 뒤 application 경로로 재실행합니다.

금지

  • 결제, 인보이스, 구독 컬렉션을 직접 수정하지 않습니다.
  • 실패 원인을 확인하지 않은 채 batch_size를 크게 늘려 재실행하지 않습니다.

일부 대상 처리 실패

증상

  • scheduler_run_logs.status=succeeded
  • summary.failed > 0

확인

  • status=succeeded는 실행 루프 자체가 완료되었다는 뜻인지 확인합니다.
  • invoices와 subscriptions에서 실패 대상의 다음 재시도 또는 최종 실패 처리 상태를 확인합니다.
  • providerCode와 failure summary가 반복되는지 확인합니다.

조치

  • 정책상 자동 재시도 대상이면 다음 retry schedule을 기다립니다.
  • 운영자 보정이 필요한 대상만 admin subscription adjustment 또는 결제 취소 API로 처리합니다.
  • 대량 provider 장애이면 Toss 장애 상태를 확인하고 같은 대상 중복 승인 위험을 검토합니다.

금지

  • summary.failed가 있다는 이유만으로 scheduler_run_logs.status를 failed로 수정하지 않습니다.
  • 같은 invoice 또는 billing cycle을 멱등성 없이 수동 승인하지 않습니다.

어드민 수동 스케줄러 실행 필요

증상

  • CronJob 누락 또는 일시 실패 후 운영자가 즉시 보정 실행해야 함
  • scheduler_run_logs에서 특정 jobType의 실행 공백 확인

확인

  • 수동 실행은 POST /admin/scheduler-runs가 API 서버에서 동기 실행한다는 점을 확인합니다.
  • Kubernetes Job 또는 CronJob을 생성하는 API가 아님을 확인합니다.
  • Idempotency-Key와 reason을 준비합니다.

조치

  • 어드민 콘솔 /admin/console/scheduler-runs에서 jobType, billingDate, limit, dryRun, reason을 입력해 실행합니다.
  • 실행 완료 응답의 runId와 operatorAuditId를 확인합니다.
  • 같은 요청 재시도는 같은 Idempotency-Key를 사용하고, 다른 payload에는 새 key를 사용합니다.

금지

  • 브라우저를 닫아도 서버 실행이 이미 시작된 요청을 임의로 취소됐다고 판단하지 않습니다.
  • 운영 사유 없이 수동 실행하지 않습니다.

Notification dead_letter 증가

증상

  • notification_outbox.status=dead_letter 증가
  • notification_worker_run_completed.dead_letter_count 증가

확인

  • last_error가 template 누락, required args 누락, SMTP permanent error, expires_at 경과 중 무엇인지 분류합니다.
  • notification_templates의 template_key와 required_template_args를 확인합니다.
  • SMTP 설정과 발신자 정책 변경 여부를 확인합니다.

조치

  • 템플릿 계약 오류이면 template 또는 enqueue args를 수정하고 새 알림 intent를 application 경로로 생성합니다.
  • SMTP permanent 오류이면 수신자 주소 또는 발신자 정책을 수정합니다.
  • 일시 장애는 retry_scheduled 상태와 backoff를 확인합니다.

금지

  • dead_letter 문서를 직접 pending으로 되돌리지 않습니다.
  • 만료된 인증 메일을 재발송하지 않습니다.

Webhook 지연 또는 누락

증상

  • Toss 결제 상태와 내부 payment/invoice 상태 불일치
  • webhook_events 처리 공백 또는 unfinished event 누적

확인

  • Toss-Signature 또는 webhook secret 검증 실패가 있었는지 확인합니다.
  • paymentKey 기준으로 Toss 조회 결과와 내부 payment status를 대조합니다.
  • webhook_events에서 같은 provider event 중복 처리 여부를 확인합니다.

조치

  • 서명 설정 오류이면 secret을 수정하고 재배포합니다.
  • provider_payment_sync 같은 문서화된 운영 보정 경로로 내부 상태를 맞춥니다.
  • 중복 event는 replay-safe 처리 결과를 확인하고 결제 원장을 직접 수정하지 않습니다.

금지

  • 서명 검증을 끄고 webhook payload를 처리하지 않습니다.
  • Toss 조회 없이 운영자 입력만으로 외부 결제 성공을 확정하지 않습니다.

Operation lock stale 의심

증상

  • 같은 구독, invoice, payment cancel 작업이 반복적으로 lock 획득 실패
  • operation_locks.locked_until_at이 지났는데 processing 로그가 남음

확인

  • locked_until_at, fencing_token, worker_id를 확인합니다.
  • 해당 작업의 scheduler_run_logs 또는 API request_id를 추적합니다.
  • 실제 Pod가 아직 실행 중인지 Kubernetes 상태를 확인합니다.

조치

  • locked_until_at이 지나고 Pod가 종료되었으면 application job을 재실행합니다.
  • 반복 stale이면 lock_duration과 작업 duration_ms를 함께 조정합니다.
  • fencing token 불일치 write는 stale worker write로 보고 무시합니다.

금지

  • 살아 있는 Pod가 소유한 lock을 수동 삭제하지 않습니다.
  • lock 우회를 위해 같은 paymentKey 또는 invoice를 직접 provider 호출하지 않습니다.

롤백과 재실행 정책

  • API 서버 rollback은 직전 정상 이미지 tag 또는 digest로 Deployment를 되돌리고 health와 admin console을 확인합니다.
  • CronJob rollback은 CronJob template image를 직전 정상 digest로 되돌린 뒤 다음 schedule 또는 수동 Job으로 검증합니다.
  • Notification Worker rollback은 Deployment image를 되돌린 뒤 retry_scheduled와 processing stale 여부를 확인합니다.
  • scheduler_run_logs는 운영 진단용 실행 이력이며 결제 원장이나 operator_audits를 대체하지 않습니다.
  • 동일 수동 실행 요청은 같은 Idempotency-Key로 재시도하고, payload가 바뀌면 새 key를 사용합니다.
  • provider 호출이 포함된 재실행은 billing_cycle_key, invoiceId, paymentKey 기준 멱등성 상태를 확인한 뒤 수행합니다.

보안과 접근 제어

  • 어드민 콘솔과 JSON API는 관리자 access token의 roles와 permissions로 권한을 확인합니다.
  • scheduler_read, scheduler_run, audit_read, product_manage 같은 권한은 최소 권한으로 부여합니다.
  • PAYMENTS_INTERNAL_SERVICE_TOKEN, Toss secret, webhook secret, 암호화 secret은 로그와 HTML 문서에 실제 값을 남기지 않습니다.
  • operator_audits는 운영 변경과 수동 실행의 책임 추적 로그입니다. 조회성 access log는 structured application log로 남깁니다.
  • scheduler_run_logs.error에는 운영자가 재실행 여부를 판단할 수 있는 요약만 저장하고 stack trace, billingKey, provider secret 원문은 저장하지 않습니다.
  • 운영자는 MongoDB에서 결제, 인보이스, 구독 상태를 직접 수정하지 않고 문서화된 admin API와 application job을 사용합니다.
