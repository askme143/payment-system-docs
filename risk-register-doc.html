PG 승인 성공 후 내부 미반영

critical failure

토스 승인 또는 자동결제는 성공했지만 내부 payment, invoice, subscription 상태 저장이 실패하는 위험입니다.

발생 조건

PG 승인 응답을 받은 직후 DB 저장, 트랜잭션 커밋, 네트워크 응답 반환, 웹훅 처리 중 하나가 실패할 때 발생합니다.

영향

회원은 돈을 냈지만 서비스 권한이나 영수증을 받지 못하고, 재시도 화면에서 중복 결제를 유도받을 수 있습니다.

탐지

provider DONE 상태와 내부 paid 상태가 불일치하는 paymentKey/orderId를 주기적으로 대조합니다.

처리 방안

  • paymentKey 또는 invoiceId 기준으로 토스 결제 조회를 수행합니다.
  • 외부 결제 사실이 DONE이면 내부 payment와 invoice를 paid로 보정하고 구독 상태와 권한을 함께 갱신합니다.
  • 보정은 관리자 보정 API로만 수행하고 operatorAuditId, 이전/이후 상태, 근거 paymentKey를 남깁니다.
  • 이미 사용자가 재시도 결제를 완료한 경우 중복 결제 여부를 판정해 한 건은 취소/환불 플로우로 넘깁니다.

예방 장치

  • PG 호출 전후의 상태를 payment 단위로 잠그고 승인 성공 반영을 단일 트랜잭션 경계로 묶습니다.
  • payment_key sparse unique index와 order_id unique index를 둡니다.
  • 승인 응답과 웹훅은 같은 paymentKey 기준으로 멱등 처리합니다.
  • 결제 성공 후 내부 저장 실패를 복구하는 provider_payment_sync 운영 경로를 유지합니다.

일반결제 승인 API 동시 호출

critical concurrency

성공 페이지 새로고침, 더블 클릭, 모바일 재전송으로 같은 paymentId 승인 요청이 동시에 들어오는 위험입니다.

발생 조건

두 요청이 모두 payment.status=ready를 읽고 동시에 PG 승인 API 호출까지 진행할 때 발생합니다.

영향

중복 승인, 서로 다른 paymentKey 충돌, 내부 paid 상태 경합, 사용자 중복 과금이 발생할 수 있습니다.

탐지

같은 paymentId/orderId에 대해 짧은 시간 안에 confirm 요청 또는 PG 승인 호출이 2회 이상 기록되는지 모니터링합니다.

처리 방안

  • 이미 paid인 paymentId는 기존 승인 결과를 200 OK로 반환합니다.
  • 같은 paymentId에 다른 paymentKey가 들어오면 409 Conflict로 차단하고 운영 로그를 남깁니다.
  • PG에서 실제 중복 승인된 경우 paymentKey별 상태를 조회해 후속 취소/환불 대상으로 분리합니다.

예방 장치

  • paymentId 기준 조건부 업데이트로 ready -> confirming 선점에 성공한 요청만 PG를 호출합니다.
  • Idempotency-Key와 request hash를 저장해 같은 키의 재시도만 같은 결과를 반환합니다.
  • payments.order_id unique와 payments.payment_key unique를 함께 둡니다.

관련 API

  • POST /payments/confirm

하나의 구매 의도에 여러 paid 시도 발생

critical data-integrity

같은 checkoutId 아래 이전 시도와 새 재시도가 모두 paid로 확정되는 위험입니다.

발생 조건

이전 시도의 승인 응답 또는 웹훅이 늦게 도착하고, 사용자는 새 paymentId/orderId로 재시도해 성공했을 때 발생합니다.

영향

같은 상품 구매 의도에서 중복 결제가 완료되고, 주문/이용권/재고가 중복 반영될 수 있습니다.

탐지

checkoutId별 paid payment 수가 2개 이상인 케이스를 집계합니다.

처리 방안

  • 먼저 확정된 paid 시도를 최종 구매로 유지합니다.
  • 늦게 paid가 된 추가 시도는 운영 취소 후보로 표시하고 자동 취소 가능 정책이면 취소 플로우를 실행합니다.
  • 사용자 화면에는 최종 구매 1건과 추가 결제 환불 진행 상태를 분리해 표시합니다.

예방 장치

  • payments에 checkout_id 기준 paid partial unique index를 둡니다.
  • checkout 상태를 ready -> paid로 조건부 전이하고 전이 성공 요청만 주문 완료 이벤트를 발행합니다.
  • 실패 후 재시도는 새 paymentId/orderId를 쓰되 checkout 단위 성공 불변식을 항상 재검증합니다.

  • POST /webhooks/toss-payments

정기결제 배치 중복 실행

critical concurrency

cron, queue worker, 운영자 수동 실행이 같은 구독 회차를 동시에 과금하는 위험입니다.

발생 조건

동일 billingDate에 둘 이상의 배치가 같은 active 구독을 선점 없이 조회할 때 발생합니다.

영향

같은 구독 회차에 인보이스와 결제가 중복 생성되고 자동결제가 중복 승인될 수 있습니다.

탐지

동일 subscriptionId와 billing period에 pending/paid invoice가 2개 이상 생기는지 감시합니다.

처리 방안

  • 중복 인보이스 중 PG 승인 전 pending 건은 voided로 닫습니다.
  • 이미 PG 승인이 된 중복 payment는 정책에 따라 취소/환불하고 감사 로그를 남깁니다.
  • 배치 실행 로그에서 중복 실행 원인과 worker 정보를 추적합니다.

예방 장치

  • billing job 실행 락을 획득한 작업만 대상 조회를 수행합니다.
  • subscription 또는 invoice 단위 claim 필드와 lease timeout을 둡니다.
  • invoices에 subscription_id + billing_period unique index를 둡니다.
  • 자동결제 요청에 회차 단위 idempotencyKey를 사용합니다.

관련 API

  • POST /internal/subscription-billing/run

실패 인보이스 재시도 중복 과금

critical retry

예약 재시도와 운영자 수동 재시도가 같은 invoiceId를 동시에 승인하는 위험입니다.

발생 조건

invoice.status=issued, 최신 payment.status=failed, latestPayment.retry_scheduled_at 조건을 두 요청이 동시에 통과할 때 발생합니다.

영향

실패 회차 하나에 여러 자동결제 승인과 여러 성공 알림이 발생할 수 있습니다.

탐지

같은 invoiceId에 성공 payment가 2개 이상 연결되는지, 같은 retry attempt가 중복 기록되는지 감시합니다.

처리 방안

  • 먼저 성공한 결제만 invoice.status=paid로 인정합니다.
  • 나머지 성공 결제는 중복 결제로 표시하고 취소/환불 플로우로 넘깁니다.
  • 실패 응답이 뒤늦게 도착한 경우 이미 paid인 invoice 상태를 덮어쓰지 않습니다.

예방 장치

  • invoiceId 기준 issued 인보이스와 최신 failed payment에 대한 재시도 잠금 획득에 성공한 요청만 PG를 호출합니다.
  • internal-billing-retry의 Idempotency-Key를 필수로 유지하고 request hash를 저장합니다.
  • operation-locks에 subscription-retry:{invoiceId} lock_key를 사용하고, 실패 결제 조회는 payments(status, retry_scheduled_at) 인덱스를 사용합니다.

관련 API

  • POST /internal/subscription-billing/{invoiceId}/retry

취소/환불 중복 또는 외부/내부 불일치

critical retry

회원/관리자 취소 요청이 중복되거나 토스 취소 성공 후 내부 이력 저장이 실패하는 위험입니다.

발생 조건

같은 paymentId에 취소 요청이 동시에 들어오거나, 토스 취소 API 성공 후 내부 cancelHistory 저장이 실패할 때 발생합니다.

영향

중복 환불, 취소 가능 금액 초과, 내부 결제 상태와 실제 환불 상태 불일치가 발생할 수 있습니다.

탐지

토스 cancelableAmount와 내부 cancelableAmount를 대조하고, 같은 paymentId의 pending cancel이 장시간 남는지 감시합니다.

처리 방안

  • paymentKey 기준으로 토스 취소 이력을 재조회해 내부 cancelHistory를 재동기화합니다.
  • 중복 취소 성공이 확인되면 주문/권한/CS 상태를 실제 환불 금액 기준으로 보정합니다.
  • 토스 취소 실패 건은 payment_cancel_requests.status를 failed로 전환하고 재시도 가능 여부를 표시합니다.

예방 장치

  • payment_cancel_requests 컬렉션에 payment_id + idempotency_key_hash unique index를 둡니다.
  • payment 단위 lock 또는 조건부 업데이트로 cancelableAmount 검증과 pending 생성이 원자적으로 처리되게 합니다.
  • 회원 취소와 관리자 취소는 같은 내부 취소 서비스를 공유합니다.

관련 API

  • POST /payments/{paymentId}/cancel
  • POST /admin/payments/{paymentId}/cancel

상품별 활성 구독 중복 생성

high data-integrity

같은 회원이 같은 productCode에 대해 둘 이상의 활성 구독을 갖는 위험입니다.

발생 조건

구독 체크아웃 또는 확정 요청이 동시에 들어와 애플리케이션 검증만으로 중복을 막으려 할 때 발생합니다.

영향

탐지

userId + productCode 기준 active/past_due/cancel_scheduled 구독 수가 2개 이상인지 점검합니다.

처리 방안

  • 중복 구독을 운영 보정 대상으로 분리하고 실제 결제/권한 사용 이력을 기준으로 유지할 구독을 하나 선택합니다.
  • 불필요한 중복 구독은 canceled 또는 migrated 상태로 닫고 관련 인보이스를 정리합니다.

예방 장치

  • subscriptions에 user_id + product_code partial unique index를 둡니다.
  • 구독 확정 전에도 같은 productCode의 활성/대기 구독을 조건부로 선점합니다.
  • 플랜 변경은 신규 구독 생성이 아니라 기존 구독 변경 플로우로만 유도합니다.

웹훅 중복 또는 순서 역전

발생 조건

  • 내부 상태보다 오래된 이벤트는 ignored로 저장하고 최신 상태를 유지합니다.
  • 외부 상태가 내부보다 최신이면 결제, 인보이스, 구독 상태를 멱등하게 보정합니다.

예방 장치

  • webhook_events(provider, event_id) unique index를 유지합니다.
  • paymentKey + provider status timestamp 기준으로 상태 전이를 단방향으로 검증합니다.
  • 웹훅 서명 또는 시크릿 검증을 필수로 둡니다.

해지 예약과 정기 과금 경합

high concurrency

사용자의 해지 예약 처리와 정기 과금 배치가 같은 구독을 동시에 처리하는 위험입니다.

발생 조건

해지 API가 status=cancel_scheduled, nextBillingDate=null을 저장하기 전 배치가 active 구독으로 조회할 때 발생합니다.

영향

해지 예약 직후 다음 회차가 과금되거나, 재개/만료/과금 상태가 서로 덮일 수 있습니다.

탐지

처리 방안

알림 template_args 비밀 노출 또는 불안정 멱등 해시

high security

로그인 링크, reset token, OTP 같은 인증 비밀이 notification_outbox.template_args, 로그, last_error에 평문으로 남거나, randomized ciphertext로 idempotency_payload_hash를 계산해 같은 알림이 충돌하는 위험입니다.

발생 조건

인증 메일 template_args를 평문 JSON으로 저장하거나, AES-GCM처럼 매번 달라지는 ciphertext를 canonical hash 입력으로 사용할 때 발생합니다.

영향

DB 또는 로그 접근자가 일회용 인증 값을 탈취할 수 있고, 같은 idempotency_key 재시도가 잘못된 conflict로 처리되어 인증/결제 알림 발송이 불안정해질 수 있습니다.

탐지

notification_outbox와 worker 로그에서 token, otp, magicLink, resetToken 같은 민감 키가 평문으로 남는지 스캔하고, 같은 idempotency_key의 hash 충돌률을 모니터링합니다.

처리 방안

  • 노출된 인증 토큰은 즉시 revoked 또는 expired 처리하고 관련 계정 접근 로그를 확인합니다.
  • 불안정 hash로 dead_letter 또는 conflict가 발생한 item은 canonical hash 정책에 맞춰 재생성합니다.
  • last_error와 worker summary에서 복호화된 template_args를 제거합니다.

예방 장치

  • 민감 template_args는 필드 단위 encrypted value object로 저장하고 Worker가 렌더링 직전에만 복호화합니다.
  • idempotency_payload_hash는 암호화 전 canonical 값 또는 민감 값의 결정적 비가역 digest로 계산합니다.
  • 로그, 테스트 assertion, last_error, scheduler summary에는 민감 값 원문을 남기지 않습니다.

medium security

발생 조건

구현자가 Toss-Signature 또는 대체 시크릿 검증을 필수로 두지 않고 웹훅 body만 신뢰할 때 발생합니다.

처리 방안

  • 서명 없는 요청 또는 검증 실패 요청은 401/403으로 거부하고 payload를 처리하지 않습니다.

예방 장치

  • 실제 토스 설정에 맞춘 서명 헤더 또는 웹훅 시크릿을 필수 계약으로 명시합니다.
  • 사용자 쿠키나 세션이 아니라 provider 인증 수단만 웹훅 인증에 사용합니다.
