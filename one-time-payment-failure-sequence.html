일반결제 실패 및 재시도 플로우

주문 생성 실패, 사용자 취소, 결제수단 인증 실패, 승인 전 서버 검증 실패, 토스 승인 실패, 실패 후 다른 결제수단 재시도를 나눠 설명합니다.

전체 시퀀스 목록전체 API 목록API 상세 설명MongoDB 구조 문서

사용 API

POST /payments/orders

일반결제 결제창 호출 전에 내부 주문을 생성합니다.

POST /payments/confirm

토스 결제 인증 후 일반결제를 최종 승인합니다.

GET /payments/{paymentId}

내부 결제 상태와 토스 승인 결과를 조회합니다.

POST /payments/{paymentId}/auth-result

토스 결제창 인증 실패 또는 사용자 취소 결과를 프론트 실패 페이지가 서버에 기록합니다.

POST /webhooks/toss-payments

토스페이먼츠 결제 이벤트를 수신하고 내부 결제/구독 상태를 보정합니다.

상태 요약

시점구독 상태결제 상태설명
주문 생성 실패-not_created결제 시도를 만들지 않거나 실패 로그만 남깁니다.
사용자 취소 직후-failed돈은 결제되지 않았고 failure.reason=user_canceled인 실패 시도로 기록됩니다.
재시도 가능 상태-retry_available같은 checkoutId 아래 새 결제 시도를 만들 수 있습니다.
인증 실패 후-failed토스 승인이 발생하지 않았고 failure.reason=auth_failed인 실패 시도로 기록됩니다.
승인 전 검증 실패-failed금액 변조나 중복 승인 가능성을 차단하고 failure.phase=confirm, failure.reason=validation_failed를 남깁니다.
실패 결과 미보고-ready서버는 실패 원인을 확정하지 못하므로 결제 시도를 즉시 failed로 바꾸지 않습니다.
만료 정책 실행 후-expired승인되지 않은 ready 결제 시도를 expired로 정리하고 필요하면 failure.reason=auth_result_not_reported를 남깁니다.
토스 승인 실패 후-failed실패 코드, 실패 메시지, 재시도 가능 여부를 현재 결제 시도에 저장합니다.
실패 웹훅 보정 후-failed늦게 도착한 이벤트가 있어도 같은 paymentKey 기준으로 멱등하게 유지합니다.
이전 시도 실패 후-retry_available같은 구매 의도 아래 새 결제 시도를 만들 수 있습니다.
새 시도 생성 후-ready새 paymentId와 orderId로 토스 결제창을 다시 호출합니다.
재시도 성공 후-paidcheckoutId 기준 최종 구매가 완료되고 모든 시도 이력이 보존됩니다.
