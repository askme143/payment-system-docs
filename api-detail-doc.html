POST /payments/orders

프론트가 일반결제 결제창을 열기 전에 서버가 구매 가능 여부와 최종 금액을 검증하고, 구매 의도(checkoutId) 아래 새 결제 시도를 생성하는 API입니다.

Header

이름필수설명
Content-Type예application/json
Authorization아니오세션 쿠키 대신 Bearer 토큰을 쓰는 경우 사용합니다.
Idempotency-Key아니오주문 생성 버튼 중복 클릭으로 같은 주문이 여러 번 생성되지 않도록 사용합니다.

Cookie

이름필수설명
session예주문을 생성하는 회원을 식별합니다.
csrf_token아니오쿠키 기반 인증을 쓰는 경우 요청 위조를 방지합니다.

Body

{
  "items": [
    {
      "productId": "prod_report_pack",
      "quantity": 1
    }
  ],
  "successUrl": "https://example.com/payments/success",
  "failUrl": "https://example.com/payments/fail",
  "checkoutId": "chk_123"
}

Body 필드

이름필수설명
items예구매하려는 상품 목록입니다. 서버는 상품 ID와 수량만 신뢰하고 가격은 서버 데이터로 다시 계산합니다.
successUrl예토스 결제창 인증 성공 후 돌아올 프론트 URL입니다.
failUrl예토스 결제창 인증 실패 후 돌아올 프론트 URL입니다.
checkoutId아니오실패 또는 취소 후 재시도할 때 기존 구매 의도에 새 결제 시도를 추가하기 위해 전달합니다. 첫 시도에서는 생략합니다.

  "status": "ready"

Response 400

상품 ID, 수량, successUrl, failUrl 형식이 잘못된 요청

Response 409

판매 중지 상품이거나 서버 계산 금액을 확정할 수 없는 요청

처리 로직

  1. 회원 인증 후 주문 생성 권한을 확인합니다.
  2. items의 상품 ID와 수량 형식을 검증합니다. 클라이언트가 보낸 가격이나 할인 금액은 신뢰하지 않습니다.
  3. 상품 판매 상태, 재고 또는 구매 가능 정책을 확인합니다.
  4. 서버의 상품/할인/세금 정책으로 최종 amount와 orderName을 계산합니다.
  5. checkoutId가 없으면 새 구매 의도를 생성하고, checkoutId가 있으면 현재 회원의 재시도 가능한 구매 의도인지 확인합니다.
  6. 토스 orderId 규칙에 맞는 6~64자의 고유 주문번호를 생성합니다.
  7. 매 결제 시도마다 새 paymentId와 새 orderId를 생성하고 attemptNo를 증가시킵니다.
  8. checkoutId, paymentId, orderId, amount, currency, customerKey, status=ready를 저장합니다.
  9. 프론트가 토스 SDK를 호출할 수 있도록 clientKey, customerKey, orderId, amount, orderName, successUrl, failUrl을 반환합니다.
프론트는 이 API 응답의 amount와 orderId로만 토스 결제창을 호출해야 합니다.
주문 생성 단계에서는 토스 승인 API를 호출하지 않습니다.
사용자 취소, 인증 실패, 승인 실패 후 다시 결제 버튼을 누르면 기존 paymentId/orderId를 재사용하지 않고 같은 checkoutId 아래 새 시도를 생성합니다.

POST /payments/confirm

토스 결제창 인증 성공 후 프론트 성공 페이지가 호출하는 일반결제 최종 승인 API입니다.

토스페이먼츠가 이 API로 직접 POST하지 않습니다. 프론트가 successUrl의 paymentKey, orderId, amount와 보관 중인 paymentId를 조합해 호출합니다.

Header

이름필수설명
Content-Type예application/json
Authorization아니오세션 쿠키 대신 Bearer 토큰을 쓰는 경우 사용합니다.
Idempotency-Key예성공 페이지 새로고침이나 네트워크 재시도로 같은 결제가 중복 승인되지 않도록 사용합니다.

Cookie

이름필수설명
session예주문을 생성한 회원과 동일한 회원인지 검증합니다.
csrf_token아니오쿠키 기반 인증을 쓰는 경우 요청 위조를 방지합니다.

Body

{
  "paymentId": "pay_123",
  "paymentKey": "paykey_...",
  "orderId": "ord_20260608_abcd",
  "amount": 25000
}

Body 필드

이름필수설명
paymentId예서버가 주문 생성 시 발급한 내부 결제 ID입니다.
paymentKey예토스 성공 리다이렉트로 받은 결제 키입니다.
orderId예토스 성공 리다이렉트로 받은 주문번호입니다. 내부 주문의 orderId와 일치해야 합니다.
amount예토스 성공 리다이렉트로 받은 결제 금액입니다. 내부 주문 금액과 일치해야 합니다.

프론트 입력값

값출처설명
paymentKeyredirectQuery토스 결제창 인증 성공 후 successUrl 쿼리로 전달됩니다.
orderIdredirectQuery토스 결제창 호출 시 사용한 주문번호가 successUrl 쿼리로 돌아옵니다.
amountredirectQuery토스 결제창 인증 성공 후 successUrl 쿼리로 전달된 결제 금액입니다.
paymentIdrouteStatesuccessUrl 쿼리, sessionStorage, route state 중 하나로 이어받습니다.
Idempotency-KeygeneratedByFrontend예: payment-confirm:pay_123

Response 409

이미 다른 paymentKey로 승인됐거나 취소/만료되어 승인할 수 없는 결제

처리 로직

  1. 회원 인증 후 paymentId가 현재 회원이 만든 결제인지 확인합니다.
  2. 내부 결제 상태가 ready인지 확인합니다. failed, expired 상태의 이전 시도는 승인하지 않습니다.
  3. request.orderId가 내부 orderId와 일치하는지 확인합니다.
  4. request.amount가 서버가 계산해 저장한 amount와 일치하는지 확인합니다.
  5. Idempotency-Key 또는 paymentId 기준으로 이미 성공 처리된 요청이면 기존 성공 결과를 200 OK로 반환합니다.
  6. 검증을 통과한 경우에만 토스페이먼츠 일반결제 승인 API POST /v1/payments/confirm을 호출합니다.
  7. 토스 승인 성공 시 paymentKey, approvedAt, method, receipt.url, 원본 응답 요약을 저장하고 내부 결제를 paid로 변경합니다.
  8. 토스 승인 실패 시 실패 코드, 실패 메시지, 재시도 가능 여부를 저장하고 현재 결제 시도를 failed로 변경합니다. failure.phase=confirm, failure.reason=provider_rejected 또는 provider_error를 저장합니다.
  9. 실패 후 재시도는 payments-orders에서 같은 checkoutId 아래 새 paymentId와 새 orderId를 발급해 진행합니다.
  10. 승인 결과와 웹훅은 같은 paymentKey 기준으로 멱등 처리합니다.
토스 시크릿 키는 서버에서만 사용하고 프론트 응답에 포함하지 않습니다.
토스 결제 인증은 성공했더라도 이 API가 승인에 성공하기 전까지 내부 결제는 paid가 아닙니다.
같은 checkoutId에 여러 결제 시도가 있어도 paid 상태가 되는 결제 시도는 하나만 허용합니다.

Response 404

처리 로직

  1. 회원 인증 후 paymentId가 현재 회원의 결제인지 확인합니다.
  2. 결제, 주문, 상품 표시 정보, 토스 승인 결과 요약을 조회합니다.
  3. payment.status는 ready, paid, failed, expired 중 하나로 반환합니다.
  4. 결제 성공 상태이면 approvedAt, receiptUrl, 결제수단 표시 정보를 포함합니다.
  5. 결제 실패 상태이면 failure.phase, failure.reason, providerCode, 실패 메시지, 재시도 가능 여부를 포함합니다.
  6. 사용자 취소는 status=failed, failure.phase=before_confirm, failure.reason=user_canceled로 반환합니다.
  7. 결제수단 인증 실패는 status=failed, failure.phase=before_confirm, failure.reason=auth_failed로 반환합니다.
  8. 토스 승인 실패는 status=failed, failure.phase=confirm, failure.reason=provider_rejected 또는 provider_error로 반환합니다.
  9. 프론트 실패 결과 기록이 도착하지 않아 만료된 결제 시도는 status=expired, failure.reason=auth_result_not_reported 또는 expired로 반환합니다.
  10. 재시도 가능한 상태이면 같은 checkoutId로 payments-orders를 다시 호출해 새 결제 시도를 만들 수 있도록 retry.available=true를 반환합니다.
  11. 토스 paymentKey, 영수증 URL, 마스킹된 카드 정보처럼 운영과 화면에 필요한 최소 정보만 응답합니다.
  12. 토스 시크릿 키, 카드 전체 번호, 원본 승인 응답의 민감 필드는 응답하지 않습니다.
완료 화면 새로고침, 실패 화면 복구, 마이페이지 결제 내역 조회가 같은 API를 사용할 수 있습니다.
조회 API는 상태를 임의로 성공 처리하지 않습니다. 승인 전 실패 기록은 payments-auth-result API가 담당하고, 미보고 결제 시도 만료는 별도 배치나 만료 조회 정책으로 수행합니다.
결제 취소/환불 상세는 결제 취소 API 상세가 작성된 뒤 취소 이력 필드를 확장합니다.

처리 로직

  1. 회원 인증 후 paymentId가 현재 회원의 결제 시도인지 확인합니다.

  4. code가 PAY_PROCESS_CANCELED이면 status=failed, failure.phase=before_confirm, failure.reason=user_canceled로 저장합니다.
  5. 그 외 결제창 인증 실패 코드는 status=failed, failure.phase=before_confirm, failure.reason=auth_failed로 저장합니다.
  6. providerCode와 message는 프론트 표시와 운영 분석을 위해 저장하되, 금전 상태의 최종 근거로 사용하지 않습니다.
  7. 재시도 가능 상태이므로 같은 checkoutId로 payments-orders를 다시 호출해 새 paymentId와 새 orderId를 발급할 수 있도록 응답합니다.
이 API는 토스 승인 전 실패 또는 사용자 취소를 기록합니다. paymentKey가 없으므로 토스 승인 API는 호출하지 않습니다.
프론트 실패 페이지가 이 API 호출에 실패하거나 사용자가 이탈하면 결제 시도는 ready로 남고, 만료 정책이 이후 expired로 정리합니다.
payment.status는 ready, paid, failed, expired처럼 큰 수명주기만 표현하고 세부 원인은 failure.phase와 failure.reason으로 구분합니다.
