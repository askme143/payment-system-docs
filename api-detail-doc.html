GET /plans

구독 체크아웃이나 플랜 변경 화면에서 사용자가 선택할 수 있는 공개 상품/구독 플랜 목록을 조회하는 API입니다. 판매 중인 구독상품과 활성 플랜만 반환하며, 결제 금액은 이후 체크아웃 또는 플랜 변경 미리보기 API에서 서버가 다시 계산합니다.

Header

이름필수설명
Authorization예백엔드 서버가 결제 시스템을 호출할 때 사용하는 내부 서비스 토큰입니다. 브라우저 세션이나 사용자 Bearer 토큰을 전달하지 않습니다.
X-Request-User-Id아니오백엔드 서버가 인증한 최종 회원 ID입니다. 로그인 사용자 기준 표시 보정이 필요할 때만 전달합니다.
X-Request-Id예백엔드 서버가 생성하거나 전달하는 요청 추적 ID입니다.

Response 400

처리 로직

  1. 공개 상태가 active인 구독상품과 신규 가입 가능한 active 플랜을 조회합니다.
  2. productCode가 있으면 해당 상품군으로, billingPeriod가 있으면 해당 결제 주기로 필터링합니다.
  3. X-Request-User-Id가 전달되면 해당 회원의 같은 productCode 활성 구독 여부와 플랜 변경 가능 여부를 계산해 isPurchasable 또는 unavailableReason을 보정합니다.
  4. 상품 표시 순서와 플랜 표시 순서를 적용하고, 판매 중지 또는 아카이브된 상품은 기본 응답에서 제외합니다.
  5. 금액, 통화, 결제 주기, 권한 요약, 상세 조회 URL처럼 화면 선택에 필요한 공개 필드만 반환합니다.
  6. 체크아웃 생성이나 플랜 변경 미리보기에서는 이 응답의 금액을 신뢰하지 않고 서버 저장 값을 다시 조회해 계산합니다.
비로그인 사용자도 가격표와 상품 선택 화면을 볼 수 있도록 공개 API로 둡니다. 여기서 public은 최종 사용자 인증이 선택이라는 뜻이며, 결제 시스템 호출 자체는 백엔드 서버의 내부 서비스 Authorization을 필수로 요구합니다.
관리자 초안, 판매 중지, 아카이브 상태의 상품/플랜은 관리자 API에서만 조회합니다.
할인, 쿠폰, 세금, 일할 계산 금액은 목록 API가 아니라 체크아웃 또는 변경 미리보기 API에서 확정합니다.

처리 로직

  1. planId로 플랜과 연결된 Product를 조회합니다.
  2. Product와 Plan이 공개 가능한 상태인지 확인합니다. 초안, 판매 중지, 아카이브 상태는 공개 상세에서 제외합니다.

  4. 플랜의 현재 가격, 결제 주기, 권한 정책, 화면 표시 정보를 반환합니다.
  5. 체크아웃 또는 플랜 변경 API가 호출될 때는 planId만 신뢰하고 가격과 판매 가능 상태를 서버에서 다시 검증합니다.
상세 API는 화면 안내용이며 결제 금액 확정 API가 아닙니다.
public은 최종 사용자 인증이 선택이라는 뜻입니다. 프론트는 자기 서버/BFF에 요청하고, 백엔드 서버가 내부 서비스 Authorization으로 결제 시스템 API를 호출합니다.
판매 중지된 기존 플랜을 이용 중인 회원의 과거 가격이나 권한 조회는 구독 상세 API에서 다루고, 공개 플랜 상세에서는 신규 선택 가능한 정보만 반환합니다.
unavailableReason은 화면 안내용 코드이며 서버의 체크아웃/변경 검증을 대체하지 않습니다.

연관 잠재 위험
  • checkout-multiple-paid-attempts 하나의 구매 의도에 여러 paid 시도 발생
  • idempotency-store-missing 멱등키 저장소 부재

X-Request-Id예백엔드 서버가 생성하거나 전달하는 요청 추적 ID입니다.

Response 400

Response 409

판매 중지 상품이거나 서버 계산 금액을 확정할 수 없는 요청

처리 로직

  1. 백엔드 서버가 인증해 전달한 X-Request-User-Id를 기준으로 주문 생성 권한을 확인합니다.
  2. items의 skuId와 수량 형식을 검증합니다. 클라이언트가 보낸 가격이나 할인 금액은 신뢰하지 않습니다.
  3. 상품 판매 상태, 구매 가능 정책, limited SKU의 availableStock(totalStock - reservedStock - soldStock)을 확인합니다.
  4. limited SKU는 원자적 조건 업데이트로 reservedStock을 주문 수량만큼 증가시키며, 재고가 부족하면 주문을 생성하지 않습니다.
  5. 서버의 상품/할인/세금 정책으로 최종 amount와 orderName을 계산합니다.
  6. checkoutId가 없으면 새 구매 의도를 생성하고, checkoutId가 있으면 X-Request-User-Id 회원의 재시도 가능한 구매 의도인지 확인합니다.
  7. 토스 orderId 규칙에 맞는 6~64자의 고유 주문번호를 생성합니다.
  8. 매 결제 시도마다 새 paymentId와 새 orderId를 생성하고 attemptNo를 증가시킵니다.
  9. paymentCustomers에서 토스 customerKey를 조회하거나 생성합니다.
  10. checkoutId, paymentId, orderId, amount, currency, paymentCustomerId, 예약된 skuId와 quantity, status=ready, expiresAt=createdAt+30분을 저장합니다.
  11. 프론트가 토스 SDK를 호출할 수 있도록 clientKey, customerKey, orderId, amount, orderName, successUrl, failUrl을 반환합니다.
프론트는 이 API 응답의 amount와 orderId로만 토스 결제창을 호출해야 합니다.
주문 생성 단계에서는 토스 승인 API를 호출하지 않습니다.
PAYMENT_ATTEMPT_TTL은 30분입니다. 결제 시도는 createdAt부터 30분 안에 payments-confirm 또는 payments-auth-result로 확정되어야 합니다.
주문 생성에서 잡은 재고 예약은 결제 승인 성공, 인증 실패 기록, 승인 실패, 또는 만료 정리 중 하나에서 반드시 해소해야 합니다.
사용자 취소, 인증 실패, 승인 실패 후 다시 결제 버튼을 누르면 기존 paymentId/orderId를 재사용하지 않고 같은 checkoutId 아래 새 시도를 생성합니다.

POST /payments/confirm

토스페이먼츠가 이 API로 직접 POST하지 않습니다. 프론트는 successUrl의 paymentKey, orderId, amount와 보관 중인 paymentId를 자기 서버/BFF로 전달하고, 백엔드 서버가 결제 시스템 API를 호출합니다.
연관 잠재 위험

  • checkout-multiple-paid-attempts 하나의 구매 의도에 여러 paid 시도 발생
  • webhook-out-of-order-or-duplicate 웹훅 중복 또는 순서 역전
  • idempotency-store-missing 멱등키 저장소 부재
